# ADR: Notification Service Architecture

**Status**: Accepted  
**Date**: 2026-06-17  
**Author**: Vincent Lin  

## 1. Context

This project is a Spring Boot demo notification service for backend assignment
review. It exposes REST APIs for creating, reading, updating, and deleting email
or SMS notifications.

The required technical environment is:

- Java 21 and Spring Boot
- MySQL for notification persistence
- Redis for notification cache and recent notification lookup
- RocketMQ for publishing created notification messages
- Docker Compose for local MySQL, Redis, RocketMQ nameserver, RocketMQ broker,
  and RocketMQ console
- Optional full Docker Compose runtime for the Spring Boot app and RocketMQ
  console message inspection

The implementation is intentionally small. It should be easy for a reviewer to
read and run locally without carrying a production-scale architecture that the
demo does not need.

The default runtime should match the assignment README: start dependencies with
Docker Compose and run the Spring Boot app on the host with Maven. A separate
full Docker runtime remains available for console-based message inspection.

## 2. Problem Statement

The service needs a clear architecture that satisfies the assignment requirements
without overbuilding:

- Persist every notification in MySQL as the source of truth.
- Publish created notifications to RocketMQ.
- Cache individual notifications and the 10 most recent notifications in Redis.
- Keep REST API, persistence, cache, and messaging responsibilities readable.
- Avoid duplicating a full layer folder structure for a single demo bounded
  context.
- Keep host development and full Docker runtime modes explicit because RocketMQ
  clients receive broker addresses from the nameserver.
- Keep the original `rocketmq-console-ng` image in the demo runtime because the
  assignment environment lists a RocketMQ Dashboard-compatible service on port
  `8088`.

The main risk is mixing infrastructure details into the core service logic. In
particular, RocketMQ should remain replaceable enough that the service behavior
can be tested without binding business flow to RocketMQ APIs.

## 3. Goals & Non-Goals

**Goals**

- Implement the README-defined notification API:
  - `POST /notifications`
  - `GET /notifications/{id}`
  - `GET /notifications/recent`
  - `PUT /notifications/{id}`
  - `DELETE /notifications/{id}`
- Store notification records in MySQL using the `notifications` table from
  `init.sql`.
- Use Redis as a performance cache, not as the source of truth.
- Publish created notifications to RocketMQ through an application-facing
  `NotificationPublisher` interface.
- Keep package layout simple enough for a demo project.
- Make host development the default runtime.
- Keep a full Docker runtime for console-based inspection.

**Non-Goals**

- Provide production-grade exactly-once messaging guarantees.
- Implement a transactional outbox, retry worker, dead letter queue handling, or
  consumer-side processing.
- Split the project into multiple services.
- Add authentication, authorization, multi-tenancy, or user management.

## 4. Considered Options

1. Single Spring Boot service with MySQL, Redis cache-aside, and RocketMQ adaptor.
2. Technical-layer package structure for every class category.
3. Event-first architecture with transactional outbox and retry processing.

## 5. Option Analysis

### Option A: Single service with cache-aside Redis and RocketMQ adaptor

- **Pros:**
  - Matches the assignment scope directly.
  - Keeps MySQL as the clear source of truth.
  - Lets Redis failures degrade to cache misses or warnings without blocking most
    API behavior.
  - Keeps RocketMQ details outside `NotificationService` through
    `NotificationPublisher`.
  - Keeps the package structure readable for a small demo.
- **Cons:**
  - Create depends on synchronous RocketMQ publishing, so publish failure fails
    the create request.
  - Does not provide durable retry for message delivery after DB commit.
  - Cache consistency is best effort and not strongly transactional.

### Option B: Technical-layer package structure for every class category

- **Pros:**
  - Familiar structure for large Spring applications.
  - Makes controller, domain, repository, and service roles explicit.
  - Scales better if many bounded contexts are added later.
- **Cons:**
  - Adds navigation cost for a single-feature demo.
  - Encourages copying a folder structure before the project has enough concepts
    to justify it.
  - Does not improve runtime behavior or assignment coverage.

### Option C: Event-first architecture with transactional outbox

- **Pros:**
  - Better production reliability for DB and message broker consistency.
  - Enables retry worker, monitoring, and dead letter handling.
  - Reduces the chance of losing publish intent after successful persistence.
- **Cons:**
  - Adds tables, background workers, state transitions, and operational concerns.
  - Overbuilds a demo assignment whose current goal is API correctness and
    integration with required technologies.
  - Requires more test and documentation surface than the current project needs.

## 6. Decision

Choose **Option A: Single Spring Boot service with MySQL, Redis cache-aside, and
RocketMQ adaptor**.

For the default local runtime, run MySQL, Redis, RocketMQ nameserver, and
RocketMQ broker in Docker Compose, then run the Spring Boot app on the host with
`./mvnw spring-boot:run` or `mvn spring-boot:run`. The default broker config
advertises `127.0.0.1:10911`, which is reachable by the host Java process through
Docker port publishing.

For console-based message lookup, use the separate full Docker runtime. It runs
the Spring Boot app and RocketMQ console in Docker and uses `broker.docker.conf`
so the broker advertises `rocketmq-broker:10911`. The full Docker runtime exposes
`styletang/rocketmq-console-ng:1.0.0` on host port `8088`.

The default host development runtime also starts `rocketmq-console-ng` to match
the assignment environment list, but its primary verification path is `mqadmin`.
The default broker advertises `127.0.0.1:10911` for the host Java process; a
console container can open and list topics, but broker-level message lookup is
only reliable in the full Docker runtime where the broker route is a Docker DNS
name.

The project keeps notification core classes in `com.example.demo.notification`
and reserves subpackages for concepts that benefit from separation:

- `dto` for API and message payload records.
- `exception` for error response and exception handling.
- `mq` for RocketMQ-specific publishing implementation.

`NotificationService` coordinates persistence, cache updates, and publish
requests. It depends on `NotificationPublisher`, not on RocketMQ-specific APIs.
`RocketMqNotificationPublisher` implements that interface in the `mq` package.

This balances assignment completeness with demo readability. It avoids the
overhead of a production messaging architecture while keeping the clearest
future seam for replacing or upgrading message delivery.

The two runtime modes intentionally use different broker advertised addresses.
This avoids requiring local `/etc/hosts` edits and keeps the README-style host
development flow working.

## 7. Architecture / Design Impact

- **Codebase impact:** The notification feature is a compact package with a
  small adaptor boundary for MQ. DTOs and exceptions remain separated because
  they have clear repeated roles.
- **Persistence impact:** MySQL stores all durable notification data. Hibernate
  uses `ddl-auto: none`; schema creation is handled by `init.sql`.
- **Cache impact:** Redis stores serialized `NotificationResponse` entries under
  `notification:{id}` with a 10-minute TTL, plus a `notifications:recent` list of
  recent notification IDs.
- **Messaging impact:** RocketMQ receives `NotificationMessage` payloads on the
  configured `notification-topic` with tags based on notification type.
- **Infrastructure impact:** Default Docker Compose starts MySQL on `3306`,
  Redis on `6379`, RocketMQ nameserver on `9876`, broker on `10911`, and
  RocketMQ console on `8088`. The Spring Boot app runs on the host at `8080`.
- **Full Docker impact:** `docker-compose.full.yaml` adds the Spring Boot app and
  keeps RocketMQ console on `8088`.
- **RocketMQ routing impact:** Default host development advertises
  `127.0.0.1:10911`. Full Docker advertises `rocketmq-broker:10911` so Docker
  containers can resolve the broker route.
- **CI/CD impact:** No special deployment pipeline is defined for this demo.
  Tests are mock/slice tests and do not require Docker.
- **Security / observability / logging impact:** Validation errors and not-found
  errors are mapped to structured responses. Redis failures are logged as
  warnings and treated as cache degradation. RocketMQ publish failures are
  surfaced as service-unavailable API errors.

## 8. Migration / Implementation Plan

- Keep the REST API contract aligned with `README.md`.
- Keep `notifications` table creation in `init.sql`.
- Keep application configuration in `src/main/resources/application.yaml`:
  - MySQL database: `taskdb`
  - MySQL user/password: `taskuser` / `taskpass`
  - Redis host/port: `localhost:6379`
  - RocketMQ nameserver: `localhost:9876`
  - RocketMQ topic: `notification-topic`
- Keep the default `docker-compose.yaml` aligned with host development:
  - No Spring Boot `app` service.
  - RocketMQ console service on `8088`.
  - RocketMQ broker config: `broker.conf`.
  - RocketMQ broker advertised address: `127.0.0.1:10911`.
- Keep full Docker settings in `docker-compose.full.yaml`:
  - MySQL URL: `jdbc:mysql://mysql:3306/taskdb?...`
  - Redis host/port: `redis:6379`
  - RocketMQ nameserver: `rocketmq-namesrv:9876`
  - RocketMQ broker advertised address: `rocketmq-broker:10911`
- Keep RocketMQ console in both Docker modes configured with
  `JAVA_OPTS=-Drocketmq.namesrv.addr=rocketmq-namesrv:9876`.
- Keep Docker service setup documented in `HELP.md`.
- If moving toward production reliability later, add an outbox table and worker
  behind the `NotificationPublisher` boundary.

Rollback strategy:

- If the adaptor boundary becomes unnecessary, `NotificationService` can be
  wired directly to the RocketMQ publisher again.
- If the demo grows into multiple domains, reintroduce package-level separation
  by bounded context rather than by technical layer alone.

## 9. Risks & Mitigations

- RocketMQ publish can fail after MySQL persistence succeeds -> Current behavior
  returns service unavailable and leaves the transaction subject to rollback.
  For production, add transactional outbox and retry.
- Redis cache can become stale or unavailable -> Treat Redis as non-authoritative;
  fall back to MySQL and rebuild recent cache when needed.
- Recent notification cache can miss if any cached entry is absent -> Fall back to
  MySQL query and replace the recent cache.
- Demo package structure can become crowded if more features are added -> Split
  by bounded context when a second meaningful feature appears.
- Running the app on the host while the broker advertises `rocketmq-broker` can
  fail to resolve broker routes -> Keep host development on `broker.conf` and
  full Docker on `broker.docker.conf`.
- Running console message lookup while the broker advertises `127.0.0.1` can
  fail from inside Docker -> Use full Docker mode for console message lookup.
- RocketMQ nameserver healthcheck can show false unhealthy if checked as HTTP ->
  Use a TCP healthcheck against port `9876` because it is a RocketMQ remoting
  port, not an HTTP endpoint.

## 10. Alternatives Considered (but not chosen)

- Full controller/domain/repository/service folder hierarchy was not chosen
  because it adds structure without improving this single-feature demo.
- Transactional outbox was not chosen because the assignment does not require
  production-grade message delivery guarantees.
- Redis as a primary store was not chosen because MySQL is the required durable
  source of truth.
- Splitting into multiple services was not chosen because the assignment is a
  single backend service exercise.
